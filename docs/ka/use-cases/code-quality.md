# კოდის ხარისხი

ბრძანებები, რომლებიც უშვებენ სტატიკურ ანალიზს ან აგენერირებენ
თანმიმდევრულ კოდს პროექტებზე.

## მოკლე ცნობარი

|ბრძანება|დანიშნულება|
|-|-|
|`JetBrainsCleanupCode`|ReSharper-ის `jb cleanupcode` გაშვება პროექტის solution-ზე|
|`GenerateApiRoutes`|API რაუტების C# კონსტანტების გენერაცია კონტრაქტების მეტამონაცემებიდან|

\---

## თუ გჭირდება solution-ის ფორმატირება და გასუფთავება

გამოიყენე commit-ის წინ, დიდი refactor-ის შემდეგ, ან იმპორტირებულ
კოდზე house style-ის გავრცელებისთვის.

**ნაბიჯები**

1. `Project Tools List` → `JetBrainsCleanupCode`

**რას აკეთებს**

* იძახებს JetBrains ReSharper-ის command-line code-cleanup utility-ს
(`jb cleanupcode`) პროექტის `SolutionFileName`-ის წინააღმდეგ
* იყენებს ReSharper-ის default-ებს — cleanup პროფილები კონფიგურირდება
IDE-ის დონეზე, არა ამ ხელსაწყოში

**წინაპირობები**

* JetBrains ReSharper Command Line Tools დაყენებული და `PATH`-ზე (`jb`)
* პროექტის `SolutionFileName` დაყენებულია მის პარამეტრებში
* `.sln` ფაილი არსებობს დისკზე

**მოსალოდნელი შედეგი**

* Solution-ის ფაილები ხელახლა იფორმატება და დალაგდება ReSharper-ის
წესების მიხედვით
* კონსოლში არ არის კოდის ცვლილების ანგარიში — გაუშვი `git status`,
რომ ნახო რა შეიცვალა

**ხშირი შეცდომები**

* **`SolutionFileName` გამოტოვებულია.** ბრძანება ლოგავს შეცდომას და
ჩერდება. ჯერ შეავსე ეს ველი პროექტის პარამეტრებში.
* **`.sln` არ არის დისკზე.** ხელსაწყო ამოწმებს ფაილს `jb`-ის გაშვებამდე
და ცდება თუ არ არის — ჩვეულებრივად ნიშნავს, რომ რეპო არ იყო
სინქრონიზებული.
* **დეფოლტი cleanup პროფილი არ არის.** explicit პროფილის გარეშე, `jb`
იყენებს default წესებს, რომლებიც შეიძლება განსხვავდებოდეს გუნდის IDE-ში
გამოყენებული წესებიდან. თუ output ცუდი გამოვიდა, გადაეცი
`--profile=...` წესი ხელით (ხელსაწყო არ გადასცემს).

## თუ გჭირდება API რაუტების კონსტანტების ხელახალი გენერაცია

გამოიყენე ახალი კონტროლერების დამატების ან გადარქმევის შემდეგ, ან
როცა API კონტრაქტების პროექტი დაიშორებს მის მომხმარებელ რაუტებს.

**ნაბიჯები**

1. `Project Tools List` → `GenerateApiRoutes`

**რას აკეთებს**

* კითხულობს პროექტის `ApiContractsProjectName`-ს, `RouteClasses`
ლექსიკონს და `CodeGenerateTestFolder`-ს
* ასინქრონებს შესაბამის Git რეპოზიტორიას (სუფთად checkout)
* აგენერირებს თითო `{ProjectName}ApiRoutes.cs`-ს თითო რეგისტრირებული
რაუტის კლასისთვის `{ApiContractsProjectFolder}/{Version}/Routes/`-ში
* თითოეული გენერირებული კლასი შეიცავს `Root` და `Version`
კონსტანტებს HTTP რაუტებისთვის

**წინაპირობები**

* `ApiContractsProjectName` დაყენებულია SupportTools პარამეტრებში
პროექტისთვის
* `RouteClasses` ლექსიკონი შევსებულია (აკავშირებს ლოგიკურ ჯგუფს კლასის
სახელთან)
* `CodeGenerateTestFolder` დაყენებულია გლობალურად SupportTools
პარამეტრებში
* კონტრაქტების პროექტის Git რეპო მისაწვდომია

**მოსალოდნელი შედეგი**

* გენერირებული `.cs` ფაილები კონტრაქტების პროექტის `Routes/` ფოლდერში
* ფოლდერის სტრუქტურა იქმნება თუ არ არსებობს

**ხშირი შეცდომები**

* **`CodeGenerateTestFolder` დაუყენებელია.** ბრძანება ჩავარდება output-ის
წარმოებამდე. ჯერ დააკონფიგურირე გლობალური პარამეტრი.
* **Git checkout-ის ჩავარდნა.** თუ underlying-მა სინქმა შეცდომა მისცა,
გენერაცია ჩუმად ჩერდება. შეამოწმე კონსოლის output სინქის შეცდომებზე
და დარწმუნდი რომ რეპო მისაწვდომია ხელახალი ცდის წინ.
* **ნაწილობრივი output ჩავარდნაზე.** თუ კონტრაქტების პროექტი ვერ
მოიძებნება სინქრონიზებულ რეპოში, ხელსაწყომ შეიძლება უკვე შექმნა
ცარიელი `Routes/` ფოლდერები. ეს ფოლდერები არ ისუფთავება — ხელით
წაშალე თუ ხელს უშლის.

## სტატიკური ანალიზი (ყოველთვის ჩართულია, არა ბრძანება)

`Directory.Build.props` ჩართავს შემდეგს გლობალურად; შენ არ უშვებ,
შენ პატივს სცემ:

* `SonarAnalyzer.CSharp` — აქტიური C# ანალიზატორი
* `AnalysisLevel = latest`, `AnalysisMode = All`
* `TreatWarningsAsErrors = true`,
`CodeAnalysisTreatWarningsAsErrors = true`
* `EnforceCodeStyleInBuild = true`

შედეგი: ნებისმიერი კომპილატორის warning, ანალიზატორის warning, ან
სტილის დარღვევა build-ს წყვეტს. პერ-პროექტი override-ი არ არის —
ყველა პროექტი იღებს ამ კონფიგურაციას.

იხ. [განვითარება](../development.md) workflow-ის შედეგებისთვის.

`stryker-config.json` არ არის ჩაკომიტებული, Stryker პაკეტები არ არის
მითითებული, და აპლიკაცია არ იძახებს Stryker-ს. Mutation testing
**არ არის** მიმდინარე pipeline-ის ნაწილი.
