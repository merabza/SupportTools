# მონაცემთა ბაზის ოპერაციები

ბაზის სასიცოცხლო ციკლის ბრძანებები იმ პროექტებისთვის, რომელთაც აქვთ
მონაცემთა ბაზა მიერთებული. ყველა ბრძანება ხელმისაწვდომია:
`Main Menu → Project Groups List → (ჯგუფი) → Projects List → (პროექტი) → Project Tools List`, მას შემდეგ რაც ინსტრუმენტები ჩართულია
`Select Project Allow Tools`-ის მეშვეობით.

ეს ბრძანებები ვარაუდობს, რომ პროექტს აქვს ვალიდური ბაზის შეერთების
პარამეტრები (იხ. [კონფიგურაცია](../configuration.md)).

## მოკლე ცნობარი

|ბრძანება|დანიშნულება|
|-|-|
|`CreateDevDatabaseByMigration`|ცარიელი dev ბაზის შექმნა EF მიგრაციით|
|`CorrectNewDatabase`|ახლად შექმნილი ბაზის post-processing (არასასურველი კონსტრუქციების შესწორება)|
|`RecreateDevDatabase`|Drop + Create + Correct ერთ ნაბიჯში|
|`DropDevDatabase`|dev ბაზის წაშლა|
|`AnaliseDevDatabase`|dev ბაზის სქემის ანალიზი|
|`AnaliseProdCopyDatabase`|prod-ის ასლის სქემის ანალიზი|
|`PrepareProdCopyDatabase`|prod-ის ასლის მომზადება scaffold-ისთვის|
|`ScaffoldSeederCreator`|scaffold + seed კოდის/JSON-ის გენერაცია prod ასლიდან|
|`JsonFromProjectDbProjectGetter`|ბაზის ჩანაწერების ექსტრაქცია JSON ფაილებში|
|`SeedData`|JSON ფაილებიდან dev ბაზაში მონაცემების ჩატვირთვა|

\---

## თუ გჭირდება ცარიელი dev ბაზა მიგრაციით

გამოიყენე როცა იწყებ ნულიდან, ან მიგრაციების ცვლილების შემდეგ
არსებული dev ბაზა გამოუყენებელია.

**ნაბიჯები**

1. გახსენი პროექტის მენიუ
2. `Project Tools List` → `CreateDevDatabaseByMigration`

**რას აკეთებს**

* ქმნის ცარიელ ბაზას (არსებულს არ შლის)
* იყენებს ყველა EF Core მიგრაციას
* ჩერდება `CorrectNewDatabase`-ის გამოყენებამდე

**მოსალოდნელი შედეგი**

* ახალი ბაზა კონფიგურირებულ სერვერზე
* სქემა შეესაბამება უახლეს მიგრაციას
* ყველა ცხრილი ცარიელია

**ხშირი შეცდომები**

* **ბაზა უკვე არსებობს.** ბრძანება ჩავარდება. ჯერ გაუშვი
`DropDevDatabase`, ან გამოიყენე `RecreateDevDatabase`.
* **მიგრაციები არ კომპილირდება.** EF ხელსაწყოები შეცდომას აჩვენებენ
კონსოლში. ჯერ ააწყვე პროექტი.
* **შეერთება იყურება prod-ზე.** გადაამოწმე პროექტის პარამეტრები
გაშვებამდე. ხელსაწყო დაეყრდნობა პროექტში დეკლარირებულ dev შეერთებას
— ის არ ვალიდირებს, რომ სამიზნე არ-prod-ია.

\---

## თუ გჭირდება dev ბაზის ჩამოყრა საწყის მდგომარეობაში

გამოიყენე როცა dev ბაზა ცუდ მდგომარეობაშია და გინდა ერთი ბრძანება,
რომელიც წაშლის, ხელახლა შექმნის და დააკორექტირებს მას.

**ნაბიჯები**

1. `Project Tools List` → `RecreateDevDatabase`

**რას აკეთებს** (თანმიმდევრობით)

1. შლის არსებულ dev ბაზას
2. ქმნის ცარიელ ბაზას მიგრაციით
3. იყენებს `CorrectNewDatabase`-ს

**მოსალოდნელი შედეგი**

* იგივე რაც `CreateDevDatabaseByMigration`, გარანტირებულად სუფთა

**ხშირი შეცდომები**

* **ღია შეერთებები.** თუ რომელიმე ხელსაწყო (Visual Studio, SSMS, EF)
ფლობს აქტიურ სესიას, drop ნაბიჯი ჩავარდება. ჯერ დახურე ყველა სესია.
* **შეუქცევადია.** ლოკალური მონაცემები იკარგება. drop-ის შემდეგ
აღდგენა შეუძლებელია.

## თუ გჭირდება production მონაცემების dev-ში გადატანა

მრავალნაბიჯიანი pipeline. შუალედური JSON ფაილები არის კონტრაქტი
ექსტრაქციასა და ჩატვირთვას შორის.

**ნაბიჯები**

1. მოიპოვე prod-ის ასლი (ეს ხელსაწყოს გარეთაა — გამოიყენე DBA-ის
პროცესი ან backup-ის აღდგენა)
2. `Project Tools List` → `PrepareProdCopyDatabase`
— ასუფთავებს prod-ის ასლს scaffold-ისთვის
3. `Project Tools List` → `JsonFromProjectDbProjectGetter`
— კითხულობს თითოეული ცხრილის ჩანაწერებს, წერს თითო JSON ფაილს
4. `Project Tools List` → `SeedData`
— კითხულობს JSON ფაილებს, ჩასვამს ჩანაწერებს dev ბაზაში

**მოსალოდნელი შედეგი**

* ფოლდერი `<TableName>.json` ფაილებით (მდებარეობა პროექტის
პარამეტრებიდან)
* dev ბაზა იგივე მონაცემებით, რაც prod-ის ასლი

**ხშირი შეცდომები**

* **თანმიმდევრობის დამოკიდებულება.** `SeedData` ეყრდნობა
დამოკიდებულებების გრაფს, რომელსაც `ScaffoldSeederCreator` აწარმოებს.
ჯერ ერთხელ მაინც გაუშვი scaffold ნაბიჯი ამ პროექტისთვის.
* **Identity სვეტები.** თუ prod ჩანაწერებში არის explicit ID-ები, dev
სქემამ უნდა დაუშვას `IDENTITY\\\_INSERT`. seeder ცხრილ-ცხრილ ცვლის
რეჟიმს, მაგრამ სქემის ფორმა უნდა ემთხვეოდეს.
* **დისკზე ღია ტექსტი.** JSON ფაილებში არის ნედლი მონაცემები, მათ
შორის სენსიტიური ველები. გადაიტანე output ფოლდერი რეპოს გარეთ და
git-ის გარეთ.
* **დიდი ცხრილები.** ექსტრაქტორი ცხრილებს მთლიანად მეხსიერებაში
ტვირთავს. რამდენიმე ასეულ MB-ზე დიდი ცხრილებისთვის გაფილტრე SQL
დონეზე გაშვებამდე.

## თუ გჭირდება ბაზის სქემის გაანალიზება

გამოიყენე dev-სა და prod-ს შორის drift-ის გამოსაკვლევად, ან scaffold
ხელსაწყოს ხედვის შესამოწმებლად.

**ნაბიჯები**

* dev: `Project Tools List` → `AnaliseDevDatabase`
* prod ასლი: `Project Tools List` → `AnaliseProdCopyDatabase`

**რას აკეთებს**

* კითხულობს ბაზის მეტამონაცემებს (ცხრილები, სვეტები, ტიპები, გასაღებები)
* აწარმოებს ანალიზის ანგარიშს (მდებარეობა პროექტის პარამეტრებიდან)

**მოსალოდნელი შედეგი**

* ანგარიშის ფაილი, რომელიც აღწერს სქემას ხელსაწყოს თვალით

**ხშირი შეცდომები**

* **მხოლოდ შეერთება.** ეს ბრძანებები პირდაპირ კითხულობენ ცოცხალ
ბაზიდან. ისინი არ ეყრდნობიან მიგრაციებს ან scaffold-ის შედეგებს.
ცარიელი ანგარიში ნიშნავს, რომ შეერთება ჩავარდა — შეამოწმე connection
string, არა პროექტის მდგომარეობა.

## თუ გჭირდება scaffold + seed კოდის გენერაცია prod-დან

გამოიყენე ახალი პროექტის bootstrap-ისთვის, ან prod-ში სქემის
ცვლილების შემდეგ scaffold მოდელის განახლებისთვის.

**ნაბიჯები**

1. `Project Tools List` → `ScaffoldSeederCreator`

**რას აკეთებს**

* აკეთებს scaffold-ს prod-ის ასლზე
* აგენერირებს EF მოდელის კლასებს
* აგენერირებს seed JSON ფაილებს
* აგენერირებს seeder-ის კოდს, რომელიც იყენებს იმ JSON ფაილებს

**მოსალოდნელი შედეგი**

* scaffold პროექტი კონფიგურირებულ ბილიკზე
* JSON ფაილები მის გვერდით, `SeedData`-სთვის მზად

**ხშირი შეცდომები**

* **იდემპოტენტურობა.** ხელახალი გაშვება ცვლის გენერირებულ კოდს.
ნებისმიერი ხელით ცვლილება scaffold პროექტში იკარგება. შეინახე
ცვლილებები partial-class ფაილებში გენერირებული ბილიკის გარეთ.
* **დაცული სიტყვები სვეტების სახელებში.** თუ prod-ის სვეტებს აქვს
C#-ის გასაღები სიტყვების სახელები, გენერირებული კოდი შეიძლება არ
კომპილირდეს. შეცვალე სვეტი ან გამოიყენე EF customization hooks.
* **გაშვების თანმიმდევრობა.** `SeedData` ჩავარდება, თუ
`ScaffoldSeederCreator` ერთხელ მაინც არ გაშვებულა ამ პროექტისთვის
— seeder-ისთვის საჭირო დამოკიდებულებების გრაფი scaffold ნაბიჯიდან
მოდის.
