# განთავსება

აპლიკაციის ინსტალაცია, განახლება და სერვისის მართვა — ლოკალურ მანქანაზე
ან მოშორებულ სერვერზე WebAgent-ის გავლით.

მოშორებული გზა იყენებს [WebAgent](../related-projects.md)-ს როგორც
აგენტს სამიზნე სერვერზე. ლოკალური გზა წერს პირდაპირ კონფიგურირებულ
ინსტალაციის ფოლდერში.

ეს ბრძანებები მუშაობს პროექტზე მას შემდეგ რაც სერვერი
დაკონფიგურირებულია `Project → Server Infos List → Create New Server Info`-ს მეშვეობით. ველების ცნობარისთვის იხ. [კონფიგურაცია](../configuration.md).

## მოკლე ცნობარი

|ბრძანება|დანიშნულება|
|-|-|
|`ProgPublisher`|deploy-სათვის პაკეტის აწყობა და სერვერზე ატვირთვა|
|`ProgramInstaller`|პროგრამის პაკეტის ინსტალაცია (ან ჩანაცვლება) სერვერზე|
|`ProgramUpdater`|აწყობა + ინსტალაცია ერთად|
|`AppSettingsEncoder`|`appsettings.json`-ის დაშიფვრა deploy-ისთვის|
|`AppSettingsInstaller`|დაშიფრული `appsettings.json`-ის გადაგზავნა სერვერზე|
|`AppSettingsUpdater`|შიფვრა + გაგზავნა ერთად|
|`ServiceStarter`|სერვერზე ინსტალირებული სერვისის გაშვება|
|`ServiceStopper`|გაშვებული სერვისის გაჩერება|
|`VersionChecker`|სერვისს ვერსიის შესახებ კითხვის გაგზავნა|
|`ProgRemover`|სერვერიდან პროგრამისა და მისი სერვისის წაშლა|
|`ServiceInstallScriptCreator` / `ServiceRemoveScriptCreator`|მხოლოდ install/remove სკრიპტების გენერაცია|

## ლოკალური vs. მოშორებული

თითოეული `ServerInfo` მიუთითებს `ServerData`-ის ჩანაწერზე, და `IsLocal`
ფლაგი ამ ჩანაწერზე წყვეტს ტრანსპორტს:

* **ლოკალური (`IsLocal = true`)** — ფაილები იწერება
`LocalInstallerSettings.InstallFolder`-ში, სკრიპტები გადის მიმდინარე
მანქანაზე. საჭიროებს საკმარის OS უფლებებს (სერვისის ინსტალაცია
ჩვეულებრივად საჭიროებს elevation-ს).
* **მოშორებული (`IsLocal = false`)** — ფაილები იგზავნება `WebAgentName`-ზე
HTTP-ით; WebAgent სერვერზე ახორციელებს ინსტალაციას. საჭიროებს
WebAgent-ის გაშვებას და მისაწვდომობას კონფიგურირებულ URL-ზე
ვალიდური API key-ით.

## თუ გჭირდება ახალი build-ის სერვერზე გადაცემა

ჩვეულებრივი გზა: settings-ის შიფვრა, პაკეტის გამოქვეყნება, ინსტალაცია,
სერვისის გაშვება.

**ნაბიჯები**

1. `Project Tools List` → `AppSettingsEncoder` — აწარმოებს დაშიფრულ
`appsettings.json`-ს
2. `Project Tools List` → `ProgPublisher` — აწყობს პროექტს და ტვირთავს
არტიფაქტს
3. `Project Tools List` → `ProgramInstaller` — ინსტალირებს არტიფაქტს
სამიზნე სერვერზე
4. `Project Tools List` → `AppSettingsInstaller` — გადაგზავნის დაშიფრულ
settings-ებს გაშვებული სერვისის ადგილზე
5. `Project Tools List` → `ServiceStarter` — უშვებს სერვისს
6. `Project Tools List` → `VersionChecker` — ადასტურებს, რომ ახალი
ვერსია მუშაობს

ან გამოიყენე კომბინირებული ბრძანებები: `ProgramUpdater` (publish +
install) და `AppSettingsUpdater` (encode + install).

**მოსალოდნელი შედეგი**

* ახალი build სერვერზეა
* სერვისი მუშაობს ახალი დაშიფრული settings-ით
* `VersionChecker` აცხადებს build-ის შესაბამის ვერსიას

**ხშირი შეცდომები**

* **WebAgent-ის ვერსიის შეუსაბამობა.** თუ WebAgent სერვერზე უფრო ძველია
ვიდრე SupportTools deployment კონტრაქტები, `VersionChecker` 10-ჯერ
ცდილობს (თითო 3 წმ ≈ 30 წმ) მანამდე, სანამ ჩუმად ჩავარდება. ჯერ
განაახლე WebAgent სერვერზე deploy-ის წინ.
* **AppSettings გასაღების შეუსაბამობა.** დაშიფვრის გასაღები გამოყავთ
`KeyGuidPart`-დან (პერ-პროექტი) და capitalized `ServerName`-დან. თუ
რომელიმე განსხვავდება encoder-ის მანქანასა და სერვერს შორის,
დეშიფვრა ჩავარდება გაშვებისას და სერვისი არ აეშვება გაუგებარი
შეცდომებით.
* **გამოტოვებული exchange settings.** `FileStorageNameForExchange` და
`SmartSchemaNameForExchange` სავალდებულოა გლობალურად
`SupportToolsParameters`-ში. შიფვრა ჩავარდება სწრაფად თუ რომელიმე
დაუყენებელია, თუნდაც პროექტის საკუთარი settings-ი სრულია.
* **სერვისის ანგარიშის უფლებები.** `ServiceUserName` სერვერზე უნდა
ფლობდეს უფლებებს: ფოლდერების შექმნა, სერვისების ინსტალაცია,
რეგისტრში ჩაწერა. Deploy აცხადებს წარმატებას სერვისის
რეგისტრაციამდე, შემდეგ სერვისი ვერ ეშვება.
* **Port-ისა და `ApiVersionId`-ის ურთიერთქმედება.** Health check
საჭიროებს `ServerSidePort > 0` და არა-ცარიელ `ApiVersionId`-ს. თუ
მხოლოდ ერთია დაყენებული, ოპერაცია ჩავარდება. თუ port არის `0`,
health check ჩუმად გამოტოვდება — სასარგებლოა, მაგრამ გაითვალისწინე.

## თუ გჭირდება სერვისის გაჩერება, გაშვება ან შემოწმება

გამოიყენე რეგულარული მაინტენანსისთვის, ან მუშა ვერსიის შესამოწმებლად.

**ნაბიჯები**

* გაჩერება: `Project Tools List` → `ServiceStopper`
* გაშვება: `Project Tools List` → `ServiceStarter`
* ვერსია: `Project Tools List` → `VersionChecker`

**მოსალოდნელი შედეგი**

* სერვისის მდგომარეობის ცვლილებები კონსოლში
* `VersionChecker` უკავშირდება სერვისის version endpoint-ს და ბეჭდავს
შედეგს

**ხშირი შეცდომები**

* **არ-არსებულ სერვისზე გაჩერება.** ბრძანება აბრუნებს წარმატების-მსგავს
output-ს, თუნდაც სერვისი არ იყოს ინსტალირებული. გადაამოწმე
`VersionChecker`-ით თუ ეჭვი გაქვს.
* **`VersionChecker` მოითხოვს სწორ endpoint-ს.** ის ელის
`TestTools.Controllers.TestController`-ს ან proxy version endpoint-ს.
ამ endpoint-ების გარეშე build-ი ყოველთვის აცხადებს ვერსიის წამოღების
ჩავარდნას.

## თუ გჭირდება ინსტალირებული პროგრამის წაშლა

გამოიყენე სერვისის გაუქმებისას, ან სუფთა ხელახალი ინსტალაციის წინ.

**ნაბიჯები**

1. `Project Tools List` → `ServiceStopper` (რეკომენდირებულია პირველად)
2. `Project Tools List` → `ProgRemover`

**რას აკეთებს**

* აჩერებს სერვისს თუ ის ჯერ კიდევ მუშაობს
* შლის ინსტალირებულ ფაილებს
* დაარეგისტრირებს სერვისს

**ხშირი შეცდომები**

* **ჩაკეტილი ფაილები.** თუ რომელიმე პროცესი ჯერ კიდევ ფლობს ფაილებს
ინსტალაციის ფოლდერში, წაშლა ნაწილობრივ ჩავარდება. ჯერ გააჩერე
სერვისი.
* **დარჩენილი scheduled task-ები.** ხელსაწყო შლის სერვისს, მაგრამ არა
დაუკავშირებელ scheduled task-ებს, რომლებიც იმავე binary-ებზე
იყურებიან. დაასუფთავე ხელით.

## როგორ მუშაობს AppSettings შიფვრა

დამატებითი ფონი deploy-ის ჩავარდნების debug-ისთვის.

1. კითხულობს გასაღებების სიას `project.AppSetEnKeysJsonFileName`-დან
— მხოლოდ კონფიგურირებული JSON ბილიკები იშიფრება, დანარჩენი
კოპირდება.
2. ქმნის სიმეტრიულ გასაღებს როგორც
`SHA256(KeyGuidPart + ServerName.Capitalize())`.
3. აერთიანებს წყაროს `appsettings.json`-ს user secrets-თან თუ
`.csproj` არსებობს.
4. შიფრავს მითითებულ string მნიშვნელობებს AES-256-CBC-ით და შემთხვევით
IV-ით, რომელიც წინ მიერთება Base64 დაშიფვრამდე.
5. ანიშნებს `VersionInfo.AppSettingsVersion`-ს timestamp-ით ისე, რომ
სერვერმა შეძლოს drift-ის აღმოჩენა.
6. წერს შედეგს `AppSettingsEncodedJsonFileName`-ში.

თუ ოდესმე დეშიფვრა ჩუმად ჩავარდება სერვერზე, ვერსიის ნიშანდება
პირველი რამეა შესამოწმებლად — შეუსაბამობა ნიშნავს, რომ სერვერზე
დაშიფრული ფაილი სხვა გასაღებითაა შექმნილი.
