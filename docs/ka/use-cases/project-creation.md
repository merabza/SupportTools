# პროექტის შექმნა

შაბლონებიდან ახალი პროექტების bootstrap-ი, არსებული პროექტის
კონფიგურაციის იმპორტი, და დამოკიდებულებების განახლება.

## მოკლე ცნობარი

|ბრძანება|დანიშნულება|
|-|-|
|`Create New Project` (template)|შაბლონიდან ახალი solution-ის გენერაცია|
|`Import Project`|ადრე ექსპორტირებული პროექტის ლოკალურ state-ში დარეგისტრირება|
|`Update Outdated Packages`|რეპოების სინქი და NuGet პაკეტების განახლება პროექტებში|
|`Clear All Groups All Solutions All Projects`|ყველა პროექტის ლოკალური ფოლდერების წაშლა|

\---

## შაბლონები

შაბლონის factory-ი (`AppCreatorFactory`) ამჟამად აწარმოებს ოთხ ტიპს:

|შაბლონი|შედეგი|
|-|-|
|`Console`|CLI აპლიკაცია, არჩევით ბაზით/რეპოზიტორიებით/მიგრაციებით|
|`Api`|REST API, არჩევით ბაზით, carcass framework-ით, Identity-ით, SignalR-ით|
|`Razor`|სერვერ-რენდერი ვებ აპლიკაცია (მინიმალური დამატებითი პროექტები)|
|`ScaffoldSeeder`|სიაშია მაგრამ factory-დან აბრუნებს null-ს — არ მუშაობს|

`TemplateModel` მართავს ფიჩერებს, რომლებიც გენერირებულ solution-ში
შედის: ბაზა, მენიუ, HTTPS, React frontend, carcass, Identity, ReCounter,
SignalR, FluentValidation, და სატესტო პროექტების სახელები.

\---

## თუ გჭირდება ახალი solution შაბლონიდან

გამოიყენე როცა იწყებ ახალ პროექტს ნულიდან.

**ნაბიჯები**

1. `Main Menu` → `Project Creator` ქვემენიუ → აარჩიე შაბლონი
2. შეავსე საჭირო ველები:
* პროექტის სახელი (გადაიყვანება ვალიდურ class-name-ად)
* solution ფოლდერის სახელი
* სამუშაო ფოლდერის ბილიკი (სად დაიდოს solution)
* უსაფრთხოების ფოლდერის ბილიკი (სად ინახება secrets და key მასალა)
* პროექტის ტიპი (Console / Api / Razor)
* API პროექტებისთვის ბაზით: მოკლე სახელი და db part ფოლდერის
სახელი
* indent size გენერირებული კოდისთვის
3. არჩევით შეავსე ბაზის პარამეტრები (dev + prod copy)

**მოსალოდნელი შედეგი**

* ახალი solution სამუშაო ფოლდერში
* პროექტი დარეგისტრირებულია SupportTools state-ში არჩეულ ჯგუფში
* ბაზის settings-ი ჩაწერილია, თუ მოწოდებული იყო

**ხშირი შეცდომები**

* **გამოტოვებული უსაფრთხოების ფოლდერი.** Creator ლოგავს შეცდომას და
ჩერდება, მაგრამ ხელახლა არ ეკითხება. დარწმუნდი, რომ უსაფრთხოების
ფოლდერი არსებობს და ჩაწერადია დაწყებამდე.
* **სახელის შეჯახება.** ამ სახელის უკვე არსებული პროექტი წყვეტს
შექმნას. გადარქმევა და ხელახლა ცდა.
* **API მოკლე სახელის გარეშე.** API შაბლონებს ბაზასთან ერთად ესაჭიროებათ
მოკლე სახელი და db part ფოლდერი ბაზასთან დაკავშირებული პროექტების
გენერაციისთვის. გამოტოვება — გექნება API მონაცემთა შრის scaffold-ის
გარეშე.

## თუ გჭირდება ადრე ექსპორტირებული პროექტის რეგისტრაცია

გამოიყენე როცა გადააქვს პროექტის კონფიგურაცია ახალ მანქანაზე.

ეს **არ არის Git clone**. Import კითხულობს `Export Project`-ის
(სხვა მანქანიდან) წარმოებულ JSON ფაილს და არეგისტრირებს მის
შინაარსს ლოკალურ state-ში.

**ნაბიჯები**

1. `Main Menu` → `Import Project`
2. მიუთითე ექსპორტირებული JSON ფაილზე

**რას აკეთებს**

* ახდენს JSON-ის deserialize-ს
* ვალიდირებს, რომ ამ სახელის პროექტი არ არსებობს
* წმენდს `ServerInfos`-ს და `AllowToolsList`-ს იმპორტირებულ პროექტზე
(სერვერების მინიჭებები და პერ-ხელსაწყო ფლაგები არ გადარჩება
export/import-ში)
* ამატებს პროექტს და მის გამოცხადებულ Git რეპოებს ლოკალურ
პარამეტრებში
* ინახავს ცვლილებას

**ხშირი შეცდომები**

* **სერვერების მინიჭებები იკარგება.** იმპორტი წაშლის პროექტის სერვერების
სიას. ხელახლა გააკეთე `Create New Server Info` იმპორტის შემდეგ.
* **Allow-tools სია იკარგება.** ჩართული პერ-პროექტი ხელსაწყოების სია
იწმინდება. ხელახლა დააკონფიგურირე `Select Project Allow Tools`-ით.
* **ჩუმად property-ის გადაგდება.** JSON-ში უცნობი ველები (მაგ. უფრო
ახალი export-ის ფორმატიდან) იგნორირდება warning-ის გარეშე. დარწმუნდი,
რომ იმპორტირებული settings-ი სწორად გამოიყურება.
* **Git ჩანაწერების შეჯახება.** თუ იმპორტში Git ჩანაწერი იზიარებს
key-ს არსებულთან, არსებული ჩანაცვლდება — სასარგებლოა refresh-ისთვის,
საშიში შემთხვევითი overwrite-ისთვის.

## თუ გჭირდება NuGet პაკეტების განახლება პროექტებში

გამოიყენე რეგულარული მაინტენანსისთვის დამოკიდებულებების ასაწევად.

**ნაბიჯები**

1. `Main Menu` → `Update Outdated Packages`

**რას აკეთებს**

* ჯერ ასინქრონებს ყველა Git რეპოზიტორიას (ისე, რომ განახლებები
მიმდინარე კოდზე დაჯდეს)
* გადის პროექტებზე და განსაზღვრავს დამოკიდებულებებს
* ანახლებს NuGet პაკეტის ვერსიებს თითოეულ `.csproj`-ში
* ხელახლა ასინქრონებს თითოეული პროექტის განახლების შემდეგ

ეს `.csproj`-ზე დაფუძნებულია; **არ** იყენებს `dotnet outdated`-ს, და
**არ** იყურება `Directory.Packages.props`-ში ცენტრალურად.

**ხშირი შეცდომები**

* **არ ამოწმებს `.csproj` ფაილების არსებობას.** არასწორად
კონფიგურირებული პროექტი შეიძლება იწვევდეს შეცდომებს შუა გაშვებაში;
სხვა პროექტებზე გაშვება გრძელდება, მაგრამ ის ერთი დასრულდება
ნაწილობრივ.
* **არ აქვს rollback ჩავარდნაზე.** ნაწილობრივი განახლებები რჩება თუ
პროექტი ჩავარდა. გამოიყენე Git-ის rollback თუ საჭიროა.
* **ხელახალი სინქი პროექტებს შორის.** თუ რეპოში ლოკალური დაუკომიტებელი
ცვლილებებია, განახლების შემდგომი სინქი ცდის მათ commit-ში იმავე batch
მესიჯით — დარწმუნდი, რომ working tree სუფთაა დაწყებამდე.

## თუ გჭირდება ლოკალური ფოლდერების წაშლა

გამოიყენე დისკის ადგილის გასათავისუფლებლად, ან ყველაფრის სუფთა
ხელახალი clone-ის წინ.

**ეს არის დესტრუქციული.** გადის დადასტურების გარეშე და შლის ყველა
პროექტის ფოლდერებს.

**ნაბიჯები**

1. `Main Menu` → `Clear All Groups All Solutions All Projects`

**რას აკეთებს**

* გადის ყველა პროექტზე (ან გაფილტრულ ქვეჯგუფზე)
* იძახებს `Clear One Solution`-ს თითო Git კოლექციისთვის, scaffold
seeder კოლექციების ჩათვლით
* შლის პროექტის სამუშაო ფოლდერებს
* გამორიცხავს ხელსაწყოს საკუთარ `bin/Debug` ფოლდერს

**მოსალოდნელი შედეგი**

* ყველა რეგისტრირებული პროექტის ფოლდერი დისკზე გაქრა
* SupportTools state (JSON პარამეტრების ფაილი) უცვლელია — პროექტები
რჩება დარეგისტრირებული, მზად ხელახალი clone-ისთვის `Sync All Projects All Gits V2`-ით.

**ხშირი შეცდომები**

* **არ ეკითხება.** როცა ბრძანებას ირჩევ, წაშლა მაშინვე იწყება. არ არის
"დარწმუნებული ხარ?" დაცვა.
* **ფოლდერების კონფიგურაცია მნიშვნელოვანია.** თუ პროექტის სამუშაო
ფოლდერი შეცდომით პარენტ ფოლდერზე იყურება, წაშლა შეიძლება უფრო შორს
წავიდეს. გადაამოწმე ფოლდერების ბილიკები პროექტის პარამეტრებში
გაშვებამდე.
* **`bin/Debug`-ის გამორიცხვა location-ზე დამოკიდებულია.** გამორიცხვა
მუშაობს მხოლოდ როცა ხელსაწყო გადის სტანდარტული build output-დან.
უჩვეულო გზიდან გაშვება შეიძლება დააბნიოს გამორიცხვის ლოგიკა.
